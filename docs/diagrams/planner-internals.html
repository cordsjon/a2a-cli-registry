How plan_chain() routes a goal through 475 CLIs — and the redesign that would let it act

The planner turns a stated goal into a ranked list of CLI chains by walking a capability graph. This page traces every stage of the live engine, then reveals the goal_actions redesign that failed five grounding passes — and why it was split off rather than forced.

01

The live pipeline

One call, six stages. A goal is a pair of type sets — what you have (goal_inputs) and what you want (goal_outputs). Hover any stage to expose its inner mechanism.

plan_chain(session, goal_inputs, goal_outputs, allow_side_effects=None, max_chain_depth=4, max_candidate_chains=100) → list[Chain]

_cap_index + adjacency build search.py:84–87

Load every capability row and every persisted CliEdge into an in-memory graph: adjacency[from] = [(to, via_type)].

One row per CLI: intent_tags · input_types · output_types · side_effect · confidence. Edges are typed by the artifact that flows across them (via_type), e.g. a text edge only forms when the producer's output type matches the consumer's input type.

Start selection search.py:97–100

Pick the CLIs a chain may begin at. If the goal names inputs, starts are CLIs that consume one. If goal_inputs is empty, starts are the no-declared-input CLIs (a query-only goal like "list files").

The empty-inputs branch is load-bearing: 276 live rows carry input_types="" for exactly this case. A naive consumes(c) & goal_in would make all of them unreachable when goal_in is empty — a fixed bug, guarded by a regression test.

BFS frontier search.py:103–121

From each start, breadth-first walk the graph. State is (path, visited, hops); a visited set guards cycles; depth is capped at max_chain_depth=4.

A path terminates the moment its tail produces a goal output — appended as a candidate. Otherwise it expands along the tail's outgoing typed edges. Cycle guard: if nxt in visited: continue.

Side-effect gate _hop_excluded search.py:46–71

Fail-unsafe prune on every hop. Destructive and unknown side-effects are excluded by default; an inferred (unverified) side-effect with real blast radius is also excluded unless the caller opts into its class.

_UNSAFE_DEFAULT = {destructive, unknown}. A writes-fs or network hop is excluded only when inferred and not in allow_side_effects. Confidence alone never excludes an inferred none hop.

Sort — then cap search.py:123–124

Every enumerated candidate is sorted before the cap is applied, then the top max_candidate_chains=100 are returned. Sorting before slicing is the cap-starvation fix.

sort_key = (length ↑, side_effect_count ↑, min_confidence_rank ↑, slugs). Before the fix (16798e3), the loop broke at 100 candidates before sorting, so a favorably-ranked start could be starved out by dict-iteration order.

Ranked chains out

A list of Chain objects, best first. Each carries its ordered slugs, hop trace, side-effect count, and worst-hop confidence.

Live probe: goal text → text at the default cap returns 100 chains; send_mail sits at sorted position 18 — reachable, verified by the shipped AC-02 test.

02

The side-effect ladder

Every capability declares one side-effect level. The gate reads them in severity order — this is the safety spine of the planner. Live counts are from the 475-CLI registry.

destructivedeletes / overwrites irreversibly4 CLIs
unknownunclassified — treated as dangerous112 CLIs
networkmakes outbound calls42 CLIs
externalacts on the outside world — send_mail is the only one1 CLI
writes-fswrites to the filesystem33 CLIs
nonepure — no blast radius283 CLIs

Two levels — destructive and unknown — are excluded by default regardless of confidence. The middle three are excluded only when their side-effect is inferred rather than declared. send_mail declares external, so it is never term-gated — the fact that unlocked side-effect reachability in the first place.

03

The redesign: a goal_actions dimension

A compound goal — "produce a report and email it" — is unplannable today. The producer's output and the mail action collapse onto one goal_outputs term, and the first text producer terminates the chain before send_mail is ever appended. The redesign models the terminal action as its own goal dimension.

# the new third list — the action verb the goal demands goal_inputs # what the user already has goal_outputs # the produced artifact only goal_actions # NEW — terminal action verbs (email, notify, webhook…)

§2.2 · matching

Tag-keyed, not enum-keyed

Match action terminals by their intent tags, not the side-effect enum (email/notify/webhook all collapse to external). Each verb maps to a discriminating tag.

email → {send}

§2.3 · terminal predicate

Artifact before action

The producer hop and the action hop must be distinct positions. Since send_mail both consumes and outputs text, a naive predicate would let it satisfy a compound goal in one false hop.

artifact_met ∧ action_met

§2.5 · edge synthesis

Reach the unreachable

send_mail has zero persisted incoming edges (hub down-weight). The planner synthesizes producer→terminal edges in memory, at plan time — no persisted-table change.

adjacency[P] += (P, T, hub)

§2.7 · authorization

Slug-scoped, final only

Self-authorization is scoped to the exact matched slugs and only at the final hop — never widened to the whole enum class (which would admit all 33 writes-fs CLIs mid-chain).

is_final ∧ slug ∈ action_terminals
04

The reveal — two holes that failed grounding

The design went through five Codex grounding passes. §2.6 and §2.7 were confirmed sound. Two claims were refuted against live code — each internally coherent, each false against the real substrate. This is why the redesign was split off, not shipped.

HOLE 1

Disjoint map values ≠ one verb per terminal

refuted ×2

The spec claimed: since the tag sets {send} and {notify} are disjoint, no terminal can match two verbs. That's a set-cover fallacy — disjoint map values don't stop a single multi-tagged terminal from satisfying multiple verbs.

send_mail.intent_tags = "notify,send" matches email → {send} ✓ matches notify → {notify} ✗ ← false match a notify goal would route to send_mail

The second attempt — a rarer-tag tie-break — died on live data too: in ~/.hermes/cli-registry.db, send and notify each occur on exactly one capability (send_mail itself). A 1–1 rarity tie, so the tie-break can't resolve. Two attempts, both refuted → stop and reframe.

HOLE 2

The "reuse the existing retry loop" claim

refuted, then fixed

The spec claimed the adapter could reuse its existing vocab-reject retry loop to re-infer on an unknown action verb. But that loop runs before the planner is ever called and only handles port tags — and the planner's structured error is swallowed into a generic "no healthy chain" before any caller sees the verb.

_infer_capability_tags retry @545–568 ← pre-planner, port-tags only plan_cli_chain call @899 ← error decoded here, too late _select_chain @668 ← flattens {error} → "no healthy chain"

Resolution: a new typed UnknownActionVerbError decode before flattening, plus a one-shot reinference on the planner-error path. This fold was confirmed sound by the fifth pass — the fix is real, it simply isn't the reuse the spec imagined.

05

The outcome — split, not forced

The 2-attempt rule fired on Hole 1. Rather than a third patch, the ticket was split: everything §2.2-independent shipped; the whole goal_actions dimension became its own ticket, with verb-matching flagged as the open design question.
